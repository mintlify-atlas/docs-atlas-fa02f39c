SIC-HVD is a server-rendered web application built on Laravel 12 and Filament 5. All user-facing functionality — creating requests, managing approvals, administering users — runs through a single Filament admin panel. Notifications are delivered asynchronously via a database-backed queue, and the production container uses FrankenPHP with Laravel Octane for high-throughput request handling.

## Application layers

Requests flow through a clear sequence of layers:

```
Browser
  └─▶  Filament admin panel  (/sic_hvd)
         └─▶  Filament Resources & Actions
                └─▶  Eloquent Models  (app/Models/)
                       └─▶  MySQL / SQLite database
```

**Filament admin panel** — all pages, forms, tables, and actions are defined as Filament Resources, Pages, and Widgets under `app/Filament/`. The panel ID is `sic_hvd`, which sets the URL prefix and is also used as the argument for `shield:generate`.

**Laravel controllers** — PDF export is the only route outside Filament. Two invokable controllers (`LocalCurrencyRequestPdfController`, `MemoRequestPdfController`) handle signed PDF download URLs registered in `routes/web.php`.

**Eloquent models** — all business data is accessed through typed Eloquent models in `app/Models/`. Soft deletes are enabled on every model, so records are never permanently removed. See [Data models and database schema](/reference/data-models) for the full field reference.

**Database** — migrations in `database/migrations/` define the schema. The application also uses Laravel's built-in `cache`, `jobs`, `sessions`, and `notifications` tables (all database-backed).

## Deployment stack

SIC-HVD ships as a Docker image built with a two-stage `Dockerfile`:

**Stage 1 — builder (`php:8.4-fpm-alpine`)**

- Installs system libraries (`icu-dev`, `libzip-dev`, `libpng-dev`, `mysql-dev`)
- Compiles PHP extensions: `intl`, `zip`, `pcntl`, `pdo_mysql`, `bcmath`, `gd`
- Runs `composer install --no-dev --optimize-autoloader`
- Runs `npm ci && npm run build` to produce compiled Vite assets

**Stage 2 — production (`dunglas/frankenphp:1.4-php8.4-alpine`)**

- Installs runtime PHP extensions via `docker-php-extension-installer`: `pdo_mysql`, `redis`, `intl`, `bcmath`, `gd`, `zip`, `exif`, `pcntl`, `opcache`
- Copies the fully built application from stage 1
- Starts Laravel Octane with the FrankenPHP server on port `8081`

```dockerfile
CMD ["sh", "-lc", "export FRANKENPHP_BINARY=$(command -v frankenphp || echo /usr/local/bin/frankenphp); \
  php artisan octane:start --server=frankenphp --host=0.0.0.0 --port=8081"]
```

The `SERVER_NAME=:8081` and `APP_RUNTIME=Laravel\Octane\FrankenPhp\Runtime` environment variables configure the FrankenPHP/Caddy server and tell Octane which runtime to use.

<Tip>
  In production, deploy the container behind a reverse proxy (for example, Caddy or Nginx) that terminates TLS and forwards traffic to port 8081.
</Tip>

## Queue system

All approval notifications are dispatched as queued jobs. The default `QUEUE_CONNECTION=database` stores jobs in the `jobs` table, which means you must run a queue worker alongside the application.

```bash
php artisan queue:work
```

In local development, `composer run dev` starts `php artisan queue:listen --tries=1 --timeout=0` automatically as part of the concurrent process group. In production, run the worker as a supervised background process (for example, using Supervisor or a Docker sidecar container).

<Note>
  Without a running queue worker, approval emails and in-app notifications will be queued but never delivered.
</Note>
