---
title: 'Roles and permissions in SIC-HVD'
description: 'How the three roles — super_admin, originator, and approver — control access in SIC-HVD, and how the function field positions approvers in the chain.'
---

SIC-HVD uses [Spatie Permission](https://spatie.be/docs/laravel-permission) together with [Filament Shield](https://github.com/bezhanSalleh/filament-shield) to manage access control. Every user is assigned one role, and that role determines which resources they can reach, which actions they can take, and whether they participate in the approval chain. Resource-level permissions are generated automatically from Filament resources using the `shield:generate` Artisan command.

## Roles overview

There are three roles in the system:

| Role | Who it is for | Can approve requests? |
|---|---|---|
| `super_admin` | System administrators | Yes (unrestricted access) |
| `originator` | Staff who create and submit requests | No |
| `approver` | Reviewers in the approval chain | Yes |

## Role capabilities

| Capability | `super_admin` | `originator` | `approver` |
|---|:---:|:---:|:---:|
| Create capital investment requests | ✓ | ✓ | — |
| Create memo requests | ✓ | ✓ | — |
| Edit own draft or returned requests | ✓ | ✓ | — |
| Submit requests for approval | ✓ | ✓ | — |
| Approve, return, or reject assigned requests | ✓ | — | ✓ |
| View all requests | ✓ | Own only | Assigned only |
| Manage users | ✓ | — | — |
| Manage roles and permissions | ✓ | — | — |
| Manage companies and departments | ✓ | — | — |
| Access system configuration | ✓ | — | — |

<Note>
  The default `super_admin` account is created by the database seeder with the email `admin@admin.com`. Change the password immediately after first login in a production environment.
</Note>

## The `function` field and approval chain position

Users with the `approver` role have an additional `function` field on their profile. This field contains a plain-text label that the system uses to determine where in the approval chain the user belongs.

The system performs a case-insensitive substring match on the `function` field to decide which approvers to show in each step's dropdown. The recognized function values are:

| Function label | Position in the chain |
|---|---|
| `director ejecutivo` | Step 1 — receives requests from originators |
| `supervisor` | Step 2 (first route) and step 4 (final step) |
| `gerente it sfh` | Step 2 (second route, alternative to supervisor) |
| `analista financiero` | Step 3 — reviews after supervisor or gerente IT |

<Tip>
  Enter the function label exactly as shown above. The match is case-insensitive, but the substring must be present. For example, a user with function `"Supervisor de Área"` will be recognized as a supervisor.
</Tip>

A Supervisor appears at two distinct points in the chain: once as the first-route option after the Director Ejecutivo, and once as the final approver after the Analista Financiero. The system distinguishes between these two positions automatically by inspecting the review history — a pending assignment that immediately follows an Analista Financiero is treated as the final step.

## How permissions are generated

Resource permissions (such as `view_any_local_currency_request` or `create_memo_request`) are generated by running:

```bash
php artisan shield:generate --all
```

This command creates permissions for every Filament resource and registers them in the database. Roles are then granted the appropriate permissions through the admin panel under **Shield → Roles**.

<Warning>
  If you add a new Filament resource, run `shield:generate` again to create its permissions. Users will not be able to access the new resource until their role has been granted the generated permissions.
</Warning>

## Managing users and roles

Users are managed in the admin panel under the **Users** section. When creating or editing a user, you can:

- Assign a role (`super_admin`, `originator`, or `approver`).
- Set the `function` field for approvers to place them in the correct step of the approval chain.
- Activate or deactivate accounts using soft deletes.

See [User management](/user-guide/user-management) for step-by-step instructions.
